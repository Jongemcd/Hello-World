Executive summary

Of the 2,426.7 hours of master downtime logged over the ~7-month period, only 979.0 hours (40.3%) reflects a genuine equipment or process failure. An almost equal share — 992.5 hours (40.9%) — is guard/door interlocks opening, logged as "downtime" but driven by very short, very frequent operator access events (53,089 of them, ~67 seconds each on average), not breakdowns. The rest is pre-warnings (10.8%), operator/procedural messages (4.4%), and changeover/recipe mismatches (3.5%).

Once that noise is removed, the picture changes completely: @Fault_0081 / @Fault_0085 ("waiting for Schubert") is not just one line item among many — at 221.1 hours it is larger than the next nine real technical fault families combined (160.3 hrs). The master-child hand-off is, by a wide margin, the single biggest true reliability problem on this line, which is exactly why the linked root-cause breakdown in this report (and the dashboard built earlier) matters most.

Classification method (so you can adjust it)

Every alarm/fault text was pattern-matched (case-insensitive) into the first bucket it hits, in this order. Anything left over defaults to Technical Failure — the model only removes what it can positively identify as noise.

One data-quality note: 37,900 master rows (about 10% of all stopped-state rows) carry a raw diagnostic string instead of a proper @Fault_xxxx code (e.g. "2A03-1K1 Profibus fault," "PacDrive fault") — these are real technical faults but were never mapped into the standard reason-code taxonomy. They're counted as Technical Failure here but flagged separately as action item 8.
