HU3 Continuous Improvement Report: True First-Fault Analysis

Revised version of the HU3 report. The first pass classified noise vs. real failure per logged row; this pass fixes a deeper problem — many alarms (like the "main air valve pressure not present" fault that led the first report) never actually start a stoppage. They fire as a downstream consequence while the machine is already stopped for another reason, then get swept up in the same row-by-row count. This version identifies each stoppage episode (from the moment MachineStatus flips 1→0 to the moment it flips back to 1) and attributes the whole episode to whichever fault was active at the exact moment it started — everything else that fires mid-episode is a cascade, not a cause.

Executive summary

Re-scoped to true stoppage episodes, HU3 logged 16,179 stoppages totaling 1,143.3 hours from Mar 11 to Jul 16, 2026. Of that, 634.5 hours (55.5%) is attributable to a confirmed technical root cause — down from the 78.9% the first pass reported, because that figure was crediting "Technical Failure" every time a real-sounding alarm fired mid-episode, even when something else (a guard, a stop button, an advisory) had actually triggered the stop.

The former #1 issue is gone entirely: "main air valve pressure not present" was the true first fault in zero of its 9,525 occurrences — it is a 100% cascade alarm. The real #1 is material exhaustion — "End of reel" plus "All rolls empty" together are 128.5 hours (20.3% of confirmed technical downtime) — followed by reject-handling capacity, pickup-valve sensing, and fork-position limit switches. Separately, a large chunk of episodes (18.7%, 214.3 hrs) start with a UPS-battery or "SR Dependency list" advisory as the logged trigger — larger than any confirmed technical cause, and flagged below for validation rather than taken at face value.

What changed, and why it matters

The first HU3 report summed each logged row's own duration and classified each row on its own alarm text. That over-credits any alarm that happens to be logged frequently while the machine is already stopped for something else. Concretely: of 9,525 "main air valve pressure not present" rows, every single one occurred while MachineStatus was already 0 — none coincided with the 1→0 transition that actually starts a stoppage. The likely mechanism: once the machine stops for any reason, the pneumatic valve stops cycling, and the pressure sensor reads "not present" as a side effect — a symptom of being stopped, not a cause of it.

This version instead groups every stopped row into its stoppage episode (the run of consecutive MachineStatus = 0 rows between one 1→0 transition and the next 0→1), takes the fault text logged at the transition itself as that episode's true first fault, and attributes the entire episode's duration to that one fault — same 4-hour cap as before, now applied per episode rather than per row, so multi-day episodes (weekends, extended planned stops) don't inflate whatever fault happened to be active when they began. On average each episode carries 2.7 cascade rows before the machine restarts; those are preserved in the supporting data as diagnostic context, not counted as separate causes.

Every stoppage episode was classified by its true first fault's text using the Warning_/Alarm_/Fault_ severity tag plus content-based overrides (rules in the Methodology section below). Only "Technical Failure" belongs in a reliability Pareto.

Statistical significance: proving the correction, not just arguing it

HU3 is a single machine, not a linked pair — so there's no second dataset to test alarms against, the way the packaging-line report tests child alarms against master stoppages. The equivalent question here is just as testable, though: the whole case for demoting "main air valve pressure not present" to a cascade artifact rests on the claim that it's a byproduct of being stopped for a while, not a specific trigger. That claim makes a clear, falsifiable prediction — episodes containing that cascade alarm should run measurably longer than episodes that don't. Below, that prediction is tested directly, plus a formal check that the six noise/real categories are genuinely different populations rather than an arbitrary split.

As with the packaging-line report, this covers the complete logged history, not a sample — so every test here asks "could a pattern this strong appear if the two things being compared were actually unrelated," not "would this hold up in a different sample." Distributions this skewed (a few very long stoppages, many short ones) are compared with Mann-Whitney U / Kruskal-Wallis rather than a t-test, which assumes normality these data don't have.

Episodes that ever show "main air valve pressure not present" run a median of 129.2 seconds — 3.6× longer than the 36.1-second median for episodes that never do (means: 402.1s vs. 101.7s). A one-sided Mann-Whitney U test confirms this isn't noise (p effectively zero at this sample size). Separately, the total number of cascade alarms logged in an episode correlates positively with how long it runs (Spearman ρ = 0.51, p ≈ 0) — exactly the pattern predicted if this alarm accumulates the longer the machine sits stopped, rather than firing because of what stopped it. Both results are consistent with the mechanism proposed in the correction above: the valve simply isn't cycling while anything is stopped, and the longer that goes on, the more likely the pressure sensor gets logged as "not present."

Are the six noise/real categories actually different populations?

A Kruskal-Wallis test across all six episode categories rejects the idea that they share one duration distribution (H = 583.6, p = 7.2×10⁻¹²⁴, n = 16,179) — confirming the classification is separating genuinely different kinds of stoppages, not just relabeling the same one. Classification method (so you can adjust it)

Step 1 — find the true first fault. Rows are grouped into stoppage episodes (consecutive MachineStatus = 0 rows bounded by a 1→0 transition and the next 0→1). Only the fault text logged at the 1→0 transition is classified and counted; every other row in the episode is a cascade, kept in the supporting data but not scored.

Step 2 — classify that fault. HU3's fault codes already carry a severity prefix — Warning_, Alarm_, or Fault_ — assigned by the control system itself. That prefix is used as the default signal (Warning → advisory/noise, Alarm/Fault → real), but content-based overrides run first and win regardless of prefix, since the same physical condition can be logged under any of the three severities depending on how long it persisted.
